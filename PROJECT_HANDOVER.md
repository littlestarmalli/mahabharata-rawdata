# Mahabharata Knowledge Graph — Project Documentation

## What This Project Is

This project takes a single **57 MB PDF file** ("The Mahabharata Set of 10 Volumes.pdf" — 3,723 pages, Hindi text with English footnotes) and extracts a complete, structured knowledge graph from it. The output is **3 interconnected JSON files** (characters, locations, timeline) plus extracted raw text, all generated by a single Python script.

---

## Input

| Item | Details |
|------|---------|
| **Source PDF** | `The Mahabharata Set of 10 Volumes.pdf` — 57 MB, 3,723 pages |
| **Content** | 10 volumes of the Mahabharata in Hindi with English footnotes |
| **Schema spec** | `charactor-creator.md` — defines the target 3-file JSON schema |

---

## Outputs

| File | Size | What It Contains |
|------|------|------------------|
| `mahabharata.py` | 181 KB, 3,576 lines | The entire pipeline — run this one file to regenerate everything |
| `characters.json` | 163 KB | 525 characters with relationships, skills, weapons, traits, lineage |
| `locations.json` | 99 KB | 52 locations with real-world coordinates and km distances |

| `map.html` | 17 KB | Interactive web map showing all 47 physical locations on India map |

```bash
cd "C:\mahabharata_hindi\The Mahabharata Set of 10 Volumes_chocr.html"
```

The `map.html` is a standalone file — just open it in a browser. It does NOT get regenerated by the script; it was hand-built separately.

## Pipeline Architecture (mahabharata.py)

The script runs **4 main steps**, and Step 4 has **8 internal phases**:

### STEP 1: PDF Extraction (line ~511)
- Uses PyMuPDF (`fitz`) to parse all 3,723 pages
- Detects volume boundaries from page headers
- Separates each volume into 3 files:
  - `volume_X_chapters.txt` — main Hindi text
  - `volume_X_footnotes.txt` — English footnotes (smaller font, bottom of page)
  - `volume_X_toc.txt` — table of contents
- Font-size threshold distinguishes body text from footnotes
- Handles multi-column layouts and page-spanning paragraphs

### STEP 1B: Manual OCR Fixes (line ~570)
- Fixes known OCR errors that affect footnote matching
- Pattern-based corrections (e.g., broken footnote numbers, merged lines)
- ~50 hardcoded fixes for specific volumes

### STEP 2: Add Section Headers to Footnotes (line ~625)
- Scans chapter text for section/parva headers
- Injects matching headers into footnote files so footnotes can be traced back to their source section
- Handles Hindi section names (e.g., "आदिपर्व", "सभापर्व")

### STEP 3: Verify Footnote Matching (line ~743)
- Cross-references every footnote number in the chapter text against the footnote files
- **Achievement: 100% match rate — 16,734 out of 16,734 footnotes matched**
- Reports per-volume statistics
- Any mismatch would indicate an extraction or OCR error

### STEP 4: Build Knowledge Graph (line ~863)

This is the main intelligence step — 8 phases:

#### Phase 1: Core Characters (line ~993)
- **~213 hardcoded `add()` calls** defining character relationships
- Covers all major and supporting characters with:
  - Father, Mother, Spouse(s), Children
  - All 101 sons of Dhritarashtra (Duryodhana through Durmarshana)
  - All 5 Pandavas, Draupadi's 5 sons, Abhimanyu's line
  - Divine births (e.g., Karna from Surya, Pandavas from various gods)
- **~35 major characters** have expanded fields:
  - `Skills` (e.g., Arjuna → ["Archery - Greatest archer", "Ambidextrous warfare", ...])
  - `Divine_Weapons` (e.g., ["Pashupatastra from Shiva", "Gandiva bow from Agni"])
  - `Titles`, `Traits`, `Character_Arc` (narrative progression)
  - `Key_Relationships` (e.g., [{With: "@krishna", Type: "Divine guide and charioteer"}])
  - `Important_Locations` (linked to location IDs)

#### Phase 2: Auto-extraction from Text (line ~1875)
- Regex-based extraction of additional character names from:
  - Footnotes (English names with descriptions)
  - Chapter text (Hindi names using Devanagari patterns)
- Extracts relationships from text patterns like "son of", "wife of", "daughter of"
- Catches characters missed by the hardcoded list

#### Phase 3: Merge & Resolve (line ~2140)
- Deduplicates characters (same person with different spellings)
- Resolves conflicting relationship data (hardcoded always wins over auto-extracted)
- Assigns gender based on relationship context (if wife → Female, if father → Male)
- **`ensure()` function**: creates minimal entries for referenced-but-undefined characters

#### Phase 3b: Auto-extract Duty/Caste (line ~2273)
- Scans text for caste/duty markers and assigns to characters
- Pattern matching for role descriptions in footnotes

#### Phase 4: Output characters.json (line ~2301)
- Builds final JSON structure with all fields
- **Auto-derives Lineage**: walks the father chain to find Grandfather and Great_Grandfather (160 characters get this automatically)
- Spouse output format: `{"@spouse_key": {"Relation": "Wife/Husband", "Children": [...]}}`
- Removes Siblings field (can be derived from shared parents) — this reduced file from 322KB to 163KB

#### Phase 5: Locations (line ~2463)
- **52 locations hardcoded** with full metadata:
  - 20 Kingdoms, 7 Cities, 2 Villages, 5 Forests, 6 Rivers, 6 Mountains, 1 Battlefield, 1 Hermitage, 4 Celestial
- Each location has: Name, Type, Region, Sub_Locations, Ruler, Famous_For, Residents, Geography
- **47 locations mapped to real-world coordinates** (lat/lng based on archaeological/historical evidence)
- **5 mythical locations** (Svarga, Brahmaloka, Patala, Amaravati, Mount Meru) marked as `Mythical: true`
- **Haversine distance computation**: calculates km distance to every other location within 500 km, stored in `Nearby_Distances_km`
- Examples: Hastinapura→Indraprastha = 97 km, Kurukshetra→Dvaitavana = 6 km

#### Phase 6: Timeline (line ~3024)
- **58 events** spanning 4 eras:
  - Pre-War: 27 events (Birth of Bhishma through War preparations)
  - Exile: 10 events (Dice game through Return from exile)
  - War: 9 events (Day 1 through Ashwatthama's night raid)
  - Post-War: 12 events (Yudhishthira's coronation through Svargarohanika)
- Year system: **0 = Kurukshetra War**, negative years before, positive after
- Range: Year -80 (Birth of Bhishma) to Year +36 (Ascent to Heaven)
- Each event has: Location (@loc_id), Participants ([@char_ids]), Outcome, Consequences

#### Phase 7: Cross-linking (line ~3459)
- Adds `Events_Occurred` to each location (which timeline events happened there)
- Adds `Major_Events` to each character (which events they participated in)
- All references use `@` prefix keys: `@char_key`, `@loc_key`, `@evt_key`

#### Phase 8: Validation (line ~3492)
- Validates ALL cross-references across all 3 files
- Checks: every @character reference exists, every @location reference exists, every @event reference exists
- Verifies timeline order continuity (Order 1..58 is continuous)
- Verifies year monotonicity (years never decrease)
- **Result: Zero validation warnings**

## Cross-Reference System

All three files are interconnected using `@` prefix keys:

| From | Field | Links To |
|------|-------|----------|
| Character | `Father`, `Mother`, `Spouse` | Other characters (`@char_key`) |
| Character | `Kingdom`, `Important_Locations` | Locations (`@loc_key`) |
| Character | `Major_Events` | Timeline (`@evt_key`) |
| Location | `Ruler`, `Residents` | Characters (`@char_key`) |
| Location | `Events_Occurred` | Timeline (`@evt_key`) |
| Timeline | `Participants` | Characters (`@char_key`) |
| Timeline | `Location` | Locations (`@loc_key`) |
| Timeline | `Related_Events` | Other events (`@evt_key`) |

The validation in Phase 8 ensures **every single reference** resolves to an existing entry. Zero broken links.

- 47 physical locations plotted as colored circle markers
- Color-coded: Red=Kingdom, Yellow=City/Village, Green=Forest, Cyan=River, Purple=Mountain, Orange=Battlefield

## Important Technical Notes

- **All character data is hardcoded** (user explicitly chose this over AI extraction for accuracy)
- The `add()` function handles deduplication — calling it twice for the same character merges data
- The `ensure()` function creates minimal stubs for characters referenced but not fully defined
- Spouse model: `list[tuple(str, str)]` internally — `(spouse_key, relation_type)`
- Lineage (Grandfather/Great_Grandfather) is auto-derived by walking the Father chain — not hardcoded
- The `@` prefix in keys is a convention for cross-reference IDs (not Python syntax)
- Distance computation uses haversine formula with R = 6,371 km; threshold 500 km for "nearby"
- PowerShell in Windows has issues with f-strings containing quotes — use helper .py files for complex commands
